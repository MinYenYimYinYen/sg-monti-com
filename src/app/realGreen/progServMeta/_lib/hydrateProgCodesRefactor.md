### Refactor Summary: Legacy vs. New Architecture

| Problem / Requirement | Legacy Solution (`progServBuilder.ts`) | New Solution (`hydrateProgCodes.ts`) |
| :--- | :--- | :--- |
| **Disjoint Data**<br>(API returns flat lists of Programs, Services, and Links) | Manually mapped IDs inside a monolithic function. | **Hydration Pipeline**. Uses dedicated helper maps (`progToServMap`, `servMap`) to join data efficiently before construction. |
| **Circular References**<br>(Program → Service → Parent Program) | **Mutation & Casting**. Created circular objects, then broke the loop by setting properties to `undefined` via `any` casting to satisfy TypeScript. | **5-Level DAG Pattern**. Uses a Directed Acyclic Graph (Stub → Leaf → Context → Rich → Root). Allows deep traversal (Service → Parent → Siblings) without infinite recursion or mutable state. |
| **Sibling Access**<br>(Service needs access to other services in the same program) | Relied on the circular reference (which was often fragile or partially broken to prevent stack overflows). | **Context ProgCodes**. The `servCode.progCode` property points to a specific "Context" parent that explicitly contains the list of sibling "Leaf" service codes. |
| **"Wrapper" Programs**<br>(Programs that exist solely as a component of another program) | `removeProgramsCodes...` function iterated nested loops to identify and filter these out. | **Ported Logic**. `filterNestedPrograms` implements the same logic but operates on the fully hydrated DAG structure to ensure "wrapper" programs don't clutter the main list. |
| **"Special" Programs**<br>(Programs consisting of a single service with the same ID) | `makeProgramSpecials` split the result into two separate arrays: `{ programCodes, specialCodes }`. | **`isSpecial` Property**. Calculated upfront and attached to the `ProgCode` object. Consumers now receive a single unified list and can filter by `isSpecial === true` if needed. |
| **Type Safety** | Used ad-hoc local types (`ProgServLink`, `ProgServParams`) that drifted from the actual data model. | **Strict Project Types**. Fully integrated with `ProgCodeWithMongo`, `ServCodeWithMongo`, and `ProgServ` to ensure consistency with the database and API contracts. |
