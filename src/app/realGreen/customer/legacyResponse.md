# Architectural Review: Legacy Customer Batch vs. Current Standards

## Executive Summary
The legacy `custBatch` logic was a robust but over-engineered solution to a complex data dependency problem (Customer $\to$ Program $\to$ Service). While it solved the "N+1" query problem and API timeout issues effectively via client-side chunking, it introduced significant technical debt through tight coupling with the React render cycle (`useEffect` orchestration).

The proposed migration to **Server-Side Orchestration** and **Slice Factories** aligns perfectly with the `sg-monti-com` architecture (Next.js 16, Redux Toolkit "Thin Slice").

## 1. Orchestration: The Shift to Server-Side
**Legacy:** Client-side State Machine (`useCustBatch` + `useEffect`).
**Critique:** Fragile. Relies on the client keeping the browser tab open and the component mounted to finish a multi-step fetch.
**Current Standard:** **Server Actions / API Routes**.
*   **Plan:** We will move the `Customer -> Program -> Service` waterfall to a Next.js API Route (or Server Action).
*   **Benefit:** The client makes *one* request ("Get Active Customers"). The server handles the complexity of fetching customers, then their programs, then their services.
*   **Optimization:** We can use `Promise.all` on the server to fetch programs and services in parallel once the customer IDs are known, or use MongoDB aggregation pipelines if the data is local, drastically reducing round-trips.

## 2. State Management: Slice Factory Pattern
**Legacy:** Single `custBatchSlice` with dynamic keys (`ACTIVE`, `PRINTED`).
**Critique:** Poor type inference, complex reducers.
**Current Standard:** **Slice Factory**.
*   **Plan:** We will implement a `createCustomerSlice(name: string)` factory.
*   **Mechanism:**
    *   Returns a standard Redux Toolkit slice.
    *   Each instance (e.g., `activeCustomerSlice`, `productionDateSlice`) is statically defined in the root reducer.
    *   **Strict Typing:** `state.activeCustomers.data` is typed, not `state.custBatch['ACTIVE'].data`.

## 3. Data Hydration: Selectors vs. Server
**Legacy:** Monolithic `buildExtCusts` builder on the client.
**Critique:** "God Object" anti-pattern. Hard to maintain.
**Current Standard:** **Composable Selectors**.
*   **Plan:** Keep the "Raw" data in Redux (normalized if possible, but flat arrays are acceptable for read-only lists).
*   **Hydration:** Use `reselect` to join data at the view layer.
    *   `selectActiveCustomers` (Base)
    *   `selectActiveCustomersWithPrograms` (Composed)
*   **Note:** If the hydration logic is purely for display (e.g., merging Tax Codes), doing it in a selector is correct. If it involves business logic (e.g., calculating totals), consider doing it on the server before sending the response.

## 4. Recursive Fetching & Chunking
**Legacy:** Recursive Thunks (`getMore`).
**Critique:** Complex client-side logic.
**Current Standard:** **Server-Side Pagination/Streaming**.
*   **Plan:** The server should handle the interaction with the upstream RealGreen API.
*   **Abstraction:** The client should not know about "chunk sizes" or "recursion depths". It should request a page or a stream.
*   **Timeout Handling:** If the dataset is massive (e.g., 5000+ customers), we will use **Streaming** (Next.js) to send data as it becomes available, rather than waiting for the full batch.

3.  **Migrate Selectors:** Port the `makeCustBatchExtCustSelector` logic to smaller, composable selectors in `src/store/selectors`.

## 2. API Architecture: The "Search Scheme" Pattern
We need a flexible system to define search dependencies (e.g., Cust -> Prog -> Serv).

### A. User Proposal (Concept)
The user proposed a type similar to `ApiContract` to define these relationships.
```typescript
export interface SearchScheme<
  TParams = CustSearch | ProgSearch | ServSearch,
  TResult = CustomerWithMongo | ServiceWithMongo | ProgramWithMongo,
  TNextParams = CustSearch | ProgSearch | ServSearch,
> {
  [key: string]: {
    params: TParams;
    result: TResult;
    toDependency: ((result: TResult) => TNextParams)
  };
}
```

### B. The "Registry" Pattern (Implementation)
Since functions (`toDependency`) cannot be serialized over the network, we will define the logic in a server-side **Registry** and reference it by name.

**`_lib/searchTypes/SearchScheme.ts`**
```typescript
import { Customer, Program, Service } from "../types";
import { CustSearch, ProgSearch, ServSearch } from "../searchTypes";

// A single step in the chain
export type SearchStep<TResult, TNextParams> = {
  entity: 'customer' | 'program' | 'service';
  // Transform the previous step's results into the next step's search params
  toParams?: (previousResult: TResult[]) => TNextParams;
};

// A full scheme is an array of these steps
export type SearchPipeline = Array<SearchStep<any, any>>;

// The Registry Type
export interface SchemeRegistry {
  [key: string]: SearchPipeline;
}
```

**`api/schemes.ts` (Server-Side)**
```typescript
export const SEARCH_SCHEMES: SchemeRegistry = {
  // Scheme: Active Customers (Cust -> Prog -> Serv)
  activeCustomers: [
    { entity: 'customer' }, // Uses initial params
    { 
      entity: 'program', 
      toParams: (customers: Customer[]) => ({ custIds: customers.map(c => c.custId) }) 
    },
    { 
      entity: 'service', 
      toParams: (programs: Program[]) => ({ progIds: programs.map(p => p.progId) }) 
    }
  ],
  // ... other schemes
};
```

### C. Vertical Streaming & Re-Render Stability
**The Challenge:** Streaming data in chunks (Vertical Chunking) means the Redux state updates frequently (e.g., every 2 seconds). If the data arrives in a random order, the UI list might "jump" or re-sort unpredictably.

**The Solution:**
1.  **Stable Sort in Selectors:** The Redux selector **MUST** enforce a strict sort order (e.g., Alphabetical by Name, or by ID). This ensures that even if "Zack" arrives before "Alice", "Alice" will slot into the correct position at the top of the list without shifting "Zack" incorrectly.
2.  **Append-Only Reducer:** The reducer should efficiently merge new items.
    *   `state.customers.push(...newCustomers)`
    *   *Optimization:* If we suspect duplicates (rare in this flow but possible), use a Map or `upsert` logic.
3.  **Virtualization:** For 2300+ items, we should use a Virtual List (e.g., `react-window`) to prevent DOM thrashing during updates.

## 4. Data Hydration (Clarification)
Hydration in this context refers to **Client-Side Subscription**.
*   **Purpose:** To link the "Dry" data (Customers, Programs, Services) fetched by this slice with "Reference" data (TaxCodes, DiscountCodes, Flags) that lives in other global slices.
*   **Mechanism:** `reselect` selectors will combine `state.activeCustomers.customers` with `state.taxCodes.data` to produce the final `ExtCust` object for the UI.
*   **Location:** This logic remains strictly in the **Selectors**, ensuring the Redux state remains normalized and the API payload remains minimal.
